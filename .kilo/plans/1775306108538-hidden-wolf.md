# Clawlet Code Review — Full Remediation Plan

Based on the comprehensive code review comparing Clawlet v0.4.7 against LangGraph, CrewAI, AutoGen, and OpenClaw. Addresses all 16 issues (A–P) organized into 4 phases.

---

## Phase 1 — Critical (Blocks correctness / causes runtime failures)

### 1.1 Issue H — Port `MemoryManager` from sync `sqlite3` to async

**Problem:** `MemoryManager` uses synchronous `sqlite3` on a `check_same_thread=False` connection, called from async code. This blocks the event loop during every `remember()`, `recall()`, `search()`, `save_long_term()`, and DB init call. `aiosqlite` is already a declared dependency but is only used in `storage/sqlite.py`.

**Files to change:**
- `clawlet/agent/memory.py` — Main rewrite target
- `clawlet/tools/memory.py` — All 9 tool `execute()` methods already async; calls become `await self.memory.xxx()`
- `clawlet/agent/loop.py:849` — `_persist_message()` already async; add `await`
- `clawlet/agent/loop.py:260` — Constructor: split into `__init__` (no I/O) + `async initialize()`
- `clawlet/agent/message_builder.py:67` — `build_messages()` is sync and calls `self.memory.get_context()`. Must become async **or** read from an in-memory cache populated during an earlier async init step.
- `clawlet/agent/identity.py:145` — Uses `get_identity_memory()` which only reads `_base_memory_content` (set from file I/O). File reads can stay sync; only DB calls need the async port. This method stays sync.
- `clawlet/runtime/assembly.py:29` — `build_runtime_services()` constructs `MemoryManager`. Must return it un-initialized, with `await mm.initialize()` called by the loop before `run()`.

**Approach:**
1. Split `MemoryManager.__init__` into `__init__` (sets paths, empty data structures, no I/O) and `async def initialize()` (opens `aiosqlite` connection, runs CREATE TABLE, loads from DB).
2. Convert `_initialize_db`, `_upsert_db_entry`, `_delete_db_entry`, `_load_long_term_from_db`, `_sync_long_term_to_db`, `search`, `recent`, `compact_long_term`, `save_long_term`, `close` to `async def`.
3. Pure-computation methods (`_is_low_value_memory`, `_sanitize_memory_value`, `_looks_like_conversation_noise`, `_fts_query`, `_dedupe_entries`) stay sync since they do no I/O.
4. `_load_long_term` (reads `MEMORY.md` via `Path.read_text`) stays sync — file I/O is acceptable in init.
5. `get_identity_memory()` stays sync (reads cached `_base_memory_content`).
6. `build_messages()` in `message_builder.py` becomes `async def build_messages()` and callers are updated.
7. Add `await memory.initialize()` in `AgentLoop.run()` alongside storage init.

**Also:** Port `clawlet/storage/sqlite.py` from fake-async `sqlite3` to real `aiosqlite` at the same time, since it has the same problem and is the pattern we're aligning to.

### 1.2 Issue D — Fix `HTTPClientManager` async lock bound to wrong event loop

**Problem:** `asyncio.Lock()` created in `__init__` is bound to the event loop active at creation time. If the singleton is created before `asyncio.run()` or the loop restarts (common in tests), the lock belongs to a dead loop and all `async with self._lock:` calls raise `RuntimeError`.

**Files to change:**
- `clawlet/providers/base.py` — `HTTPClientManager`

**Approach:**
Replace the class-level `_lock` with a lazily-created lock via a property:

```python
@property
def _async_lock(self) -> asyncio.Lock:
    loop = asyncio.get_running_loop()
    loop_id = id(loop)
    if self._current_loop_id != loop_id:
        self._lock_instance = asyncio.Lock()
        self._current_loop_id = loop_id
    return self._lock_instance
```

Update `get_client()` to use `async with self._async_lock:` instead of `async with self._lock:`.

### 1.3 Issue — Tests: Security-critical shell tool has zero tests

**Problem:** `ShellTool._validate_command()` is the primary security boundary. It has no tests. A single regression could allow arbitrary command execution.

**Files to create:**
- `tests/unit/test_shell_tool.py`

**Tests to write:**
1. **Whitelist enforcement:** Allowed commands pass; non-whitelisted commands are rejected.
2. **Dangerous pattern blocking:** Each pattern in `DANGEROUS_PATTERNS` is blocked (`;`, `&&`, `||`, `$()`, backticks, `rm -rf /`, etc.).
3. **Pipe-to-shell blocking:** `|sh`, `|bash` blocked.
4. **Inline Python validation:** Complex `-c` code blocked; simple expressions allowed.
5. **Empty command handling:** Empty string, whitespace-only string rejected.
6. **`allow_dangerous=True` bypass:** Patterns pass when flag is set, command still needs whitelist.
7. **Segment extraction:** Multi-command chaining via `&&` correctly extracts all base commands.
8. **Timeout behavior:** Command exceeding timeout returns failure (async test).

**Also create:**
- `tests/unit/test_memory_manager.py` — `remember`, `recall`, `forget`, `search`, `recent`, `compact_long_term`, `curate_from_recent_daily_notes`, `_sanitize_memory_value`, `_is_low_value_memory`.
- `tests/unit/test_rate_limiter.py` — `is_allowed`, `check_message`, `check_tool`, `_cleanup`, `TokenBucket.consume/wait_for_tokens`.
- `tests/unit/test_circuit_breaker.py` — State transitions: CLOSED -> OPEN after threshold, OPEN -> HALF_OPEN after timeout, HALF_OPEN -> CLOSED on success, execute raises `CircuitBreakerOpen` when open.

**Test infrastructure fixes:**
- Add `[tool.pytest.ini_options]` to `pyproject.toml` with `testpaths = ["tests", "clawlet/tests"]` and `asyncio_mode = "auto"`.
- Create `tests/conftest.py` with shared fixtures (tmp workspace, mock provider, mock bus).
- Update `Makefile` test targets to discover all test directories.

## Phase 2 — Medium Priority (Logic errors, DRY violations, cross-platform)

### 2.1 Issue B — Consolidate dual circuit-breaker implementations

**Problem:** `retry.py` has a proper `CircuitBreaker` class. `loop.py` reimplements an inline version with raw instance variables. The two diverge (inline version lacks `HALF_OPEN` state).

**Files to change:**
- `clawlet/agent/loop.py` — Replace `_consecutive_errors`, `_max_consecutive_errors`, `_circuit_open_until`, `_circuit_timeout_seconds` with a `CircuitBreaker` instance from `retry.py`.
- `clawlet/retry.py` — No changes needed; the class is already correct.

**Approach:**
```python
# loop.py __init__
from clawlet.retry import CircuitBreaker
self._provider_circuit = CircuitBreaker(
    failure_threshold=5,
    recovery_timeout=30.0,
    expected_exception=Exception,
)
```

Update `_call_provider_with_retry()` to use `self._provider_circuit.record_success()` / `record_failure()` / `can_execute()` instead of manual datetime comparisons.

Also do the same for the tool circuit breaker (`_tool_failures`, `_tool_circuit_open_until`, etc.) — create a second `CircuitBreaker` instance per tool or a dict of them.

### 2.2 Issue L — Fix `LLMResponse.tool_calls` mutable default

**Problem:** `tool_calls: list = None` on a dataclass. Any code doing `response.tool_calls.append(x)` without a None check will raise `AttributeError`.

**File to change:**
- `clawlet/providers/base.py:22`

**Fix:**
```python
from dataclasses import field
tool_calls: list = field(default_factory=list)
```

Then grep for `if response.tool_calls` / `if resp.tool_calls is not None` guards across the codebase and simplify them since the field is now always a list.

### 2.3 Issue E — Cross-platform shell execution

**Problem:** `shell.py:229` hardcodes `/bin/bash -lc` for `allow_dangerous` mode. On Windows (`win32`, the current dev platform), this path doesn't exist.

**File to change:**
- `clawlet/tools/shell.py`

**Approach:**
```python
import sys

if use_shell:
    if sys.platform == "win32":
        shell_cmd = ["powershell.exe", "-Command", command]
    else:
        shell_cmd = ["/bin/bash", "-lc", command]
    process = await asyncio.create_subprocess_exec(
        *shell_cmd,
        stdout=asyncio.subprocess.PIPE,
        stderr=asyncio.subprocess.PIPE,
        cwd=str(self.workspace),
    )
```

Also update `DANGEROUS_PATTERNS` with Windows-specific patterns (e.g., `del /f /s /q C:\`, `format C:`).

### 2.4 Issue N — Unify validation style (tuple vs exception)

**Problem:** `validate_tool_params()` in `registry.py` imports `ValidationError` but returns `(bool, str, dict)` tuples. `exceptions.py` has both `ValidationError` and helper functions returning tuples. Callers must know which style each function uses.

**Files to change:**
- `clawlet/tools/registry.py` — Have `validate_tool_params` raise `ValidationError` on failure instead of returning a tuple. Return the sanitized dict on success.
- `clawlet/exceptions.py` — Remove the tuple-returning helper functions (`validate_not_empty`, `validate_string_length`, `validate_type`) and replace them with functions that raise `ValidationError`.
- All callers of `validate_tool_params` — Update from `is_valid, err, sanitized = validate_tool_params(...)` to `try/except` pattern.

**Note:** This is a broader refactor. Grep for all callers first. If there are many, consider a deprecation period where both styles work.

### 3.3 Issue G — Make config reload atomic

**File to change:**
- `clawlet/config.py:769-779`

**Approach:**
Instead of mutating `self.__dict__` in place, have `reload()` return a new `Config` instance. Callers swap the reference atomically:

```python
def reload(self) -> "Config":
    """Reload configuration from the original YAML file."""
    if self.config_path and self.config_path.exists():
        new_config = Config.from_yaml(self.config_path)
        new_config.config_path = self.config_path
        return new_config
    raise FileNotFoundError("Cannot reload config: file not found")
```

Callers: `self.config = self.config.reload()` instead of `self.config.reload()`.

### 3.4 Issue F — Consolidate command validation pipeline in ShellTool

**Problem:** Dangerous patterns are checked via both regex (`DANGEROUS_PATTERNS`) and set membership (`CONTROL_OPERATORS`). The two overlap and can diverge.

**File to change:**
- `clawlet/tools/shell.py`

**Approach:**
1. Remove bare `|` from `DANGEROUS_PATTERNS` (it's covered by the set-based check).
2. Document the two layers clearly: "Layer 1: regex patterns for complex dangerous constructs. Layer 2: token-level operator detection for chaining/redirection."
3. Ensure the regex check runs first (fast reject) and the token-level check runs second (structural analysis).
